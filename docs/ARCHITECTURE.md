# Архитектура проекта Vibe

Этот документ описывает техническое устройство платформы Vibe. Проект разделен на слабо связанные Backend (REST API) и Frontend (SPA).

---

## 1. База Данных (SQLite + SQLAlchemy)

В проекте используется реляционная база данных со строгими ограничениями внешних ключей (Foreign Keys). 

### Таблицы и связи (Schema)
- **`Users` (Пользователи)**: Хранит данные для авторизации (хэш пароля) и профиль.
- **`Events` (События)**: Связана с `Users` (многие-к-одному) через `creator_id`.
- **`Participants` (Участники/Заявки)**: Промежуточная таблица (Связь многие-ко-многим между Users и Events) с дополнительным полем `status` (`pending`, `accepted`, `rejected`).
- **`Reviews` (Отзывы)**: Связывает `from_user_id` (кто оставил), `to_user_id` (кому оставили) и `event_id` (в рамках какого события). Рейтинг строго от 1 до 5 (CheckConstraint).

**Каскадное удаление (Cascade Deletes):**
При удалении пользователя каскадно удаляются все созданные им события. При удалении события — удаляются все привязанные к нему заявки (`Participants`) и отзывы (`Reviews`), чтобы избежать нарушения целостности базы данных (IntegrityError).

## 2. Backend (FastAPI)

Бэкенд построен на принципах модульности и строгой типизации:

- `backend/app/database.py` — настройка подключения к SQLite.
- `backend/app/database/` — модели данных SQLAlchemy (описание структуры БД).
- `backend/app/schemas/` — схемы Pydantic. Отвечают за сериализацию, десериализацию и валидацию данных, приходящих от клиента.
- `backend/app/routes/` — бизнес-логика и эндпоинты API. Разделены по сущностям (`users.py`, `events.py`, `auth.py`).
- `backend/app/utils/security.py` — работа с паролями (Passlib/bcrypt) и генерация JWT-токенов. Авторизация происходит через зависимость (Dependency) `get_current_user`.

---

## 3. Frontend (React.js)

SPA (Single Page Application), общающееся с сервером через Axios.

- **Стилизация:** Используется подход Vanilla CSS без тяжеловесных фреймворков. Настроен глобальный дизайн-код (CSS Variables) с эффектами Glassmorphism.
- **Контекст (Context API):** `AuthContext` отвечает за хранение JWT-токена в `localStorage`, слежение за статусом авторизации и управление видимостью глобального окна входа/регистрации (`AuthModal`).
- **Компонентный подход:** Интерфейс разбит на переиспользуемые части (`Header`, `AuthModal`, карточки событий).
- **Роутинг:** Реализован с помощью `react-router-dom`.

---

## 4. Безопасность

- Пароли никогда не хранятся в открытом виде (используется `bcrypt`).
- Все защищенные эндпоинты API требуют передачи JWT-токена в заголовке `Authorization`. Токены имеют срок годности.
- Любое критическое действие (например, удаление или изменение события) проверяет, является ли текущий пользователь создателем ресурса (`event.creator_id == current_user.id`).
- Удаление аккаунта дополнительно защищено необходимостью ввести текущий пароль пользователя в теле запроса.
